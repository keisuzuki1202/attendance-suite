# テストケース

## A. 自動テスト（`npm test` / `tests/run.ts`）— 19ケース 全合格

### 単体: 勤怠計算
- 通常勤務 9:00-18:00 休憩60 → 実働8h・残業0
- 残業 9:00-20:00 休憩60 → 実働10h・残業2h
- 深夜帯 22:00-24:00 → 120分
- 退勤<出勤 → 0扱い
- minutesToHHMM(125) === "2:05"

### 単体: RBAC / 内部統制
- 従業員は管理者設定不可 / 管理者は可
- 産業医は申請不可・閲覧のみ
- 承認者本人は自分の申請を承認不可
- isApprover 判定

### 単体: バリデーション
- 有給申請は予測効果・該当日時が必須
- 打刻修正は理由・対象が必須
- 必須充足で成功

### 結合: DB整合性（seed）
- ユーザー6名以上 / 全5ロール存在
- 提出済み申請は必ず2承認ステップ
- 承認完了申請は currentStepLevel=0
- PCログ・監査ログが記録されている

## C. 追加で確認すべき受け入れ観点（手順）

1. **2階層承認の通し**: hanako で有給申請→kacho 承認→bucho 承認→完了通知が hanako/kacho に届き、有給残が減る。
2. **差戻し→再申請**: kacho が差戻し→hanako に差戻し通知→再申請で1次から再開。
3. **打刻修正の反映**: correction 申請が2次承認完了で対象勤怠の値が更新される（非ロック時）。
4. **月次確定→ロック→産業医共有**: hanako 月次確定→kacho/bucho 電子承認→ロック→admin がレポート生成→共有→doctor が確認済み。
5. **確定後の直接編集禁止**: ロック済み日を `/attendance` で保存しようとすると 409。
6. **CSV/PDF**: 産業医レポートCSV・長時間労働者CSV・月次PDFが出力でき、DL履歴が `audit_logs` に残る。
